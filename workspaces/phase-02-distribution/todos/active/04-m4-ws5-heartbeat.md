**Settled `/todos` design decisions** (per the WS-5 deep-dive § "Open questions for `/todos`", carried into the shard scopes below): (1) **STAR single-server**, NOT Prio — native structural k-anonymity, one Foundation operator matching `specs/foundation-ops.md` registry #5, OHTTP recovers the non-collusion property at the network layer; (2) **client-side / local DP** injected per-counter BEFORE share-split; (3) **per-metric published ε** with **total-ε-over-a-stated-window** accounting (per-counter ε does NOT compose across 21 counters × M heartbeats — the privacy claim is the total, not the per-counter value); (4) **Tor opt-in (default off)** — OHTTP already strips IPs, Tor is defense-in-depth with an exit-node-MITM cost (`specs/network-security.md:77` item 4); (5) the **k≥100 floor reads the TRUE cohort count, not the noised count** (pinned below in S11 — the structural threshold reveal operates on real submitter cardinality; DP noise perturbs the per-counter VALUE, not the cohort-size gate).

**Known signal-gap to surface, NOT a leak (per `rules/communication.md` — frame in plain language):** at fewer than 100 total opted-in users the k≥100 floor makes the telemetry **structurally inert** — no aggregate is ever revealed below the threshold. This is correct privacy behavior (a sub-100 cohort is cryptographically unrecoverable, not withheld-by-policy), but it means the Foundation gets **zero health signal until the install base crosses ~100 consenting users**. The plain-language framing for the user: "Until about 100 people opt in, we deliberately learn nothing — the math refuses to reveal a number that small. The trade-off is no early-population health data; the benefit is that a small or rare cohort can never be de-anonymized." Surface this in the transparency report (withheld-cohort events are auditable per `specs/foundation-health-heartbeat.md:55`, `errors.py:55` `kAnonymityFloorViolatedError`), do NOT pad cohorts to reach the floor.

**Type:** Build+Wire

**Value-anchor:** `briefs/00-phase-02-scope.md:77-79` (source b — the active-workspace brief) reads VERBATIM: "The single largest de-scoped Phase-01 item. … OHTTP (RFC 9458) Key Configuration Server + Relay (strips source IPs) (`foundation-health-heartbeat.md:19-21`)." S10 delivers the network substrate without which no client crypto can run: per the deep-dive, "WS-5's first concrete deliverable is Foundation-ops infrastructure, not client code" — the existence-check-first prerequisite (`rules/verify-resource-existence.md` MUST-2). It is the root primitive (no deps, wave-1 per `02-plans/01-architecture.md` § Roots alongside S1/S4s/S8).

**Implements:** `specs/foundation-ops.md` registry #3 OHTTP Key Configuration Server (§19), registry #4 OHTTP Relay (§20 — "Foundation or third-party; Strips source IPs"), signing ceremonies 2-of-N Foundation stewards (§77-83 — the Key Config is an operator-signed Ed25519 artifact), `OHTTPRelayDownError` (§108); `specs/foundation-health-heartbeat.md` design stack item 3 (§21, RFC 9458); `specs/network-security.md` TLS 1.3 + cert pinning + strict SNI + HSTS for Foundation endpoints (§5,15,24).

**Depends:** — (root; wave-1, dependency-free). S11 depends on this shard; S12 depends on S11.

**Scope:** Stand up the Foundation-operated OHTTP Key Configuration Server publishing the operator-signed (Ed25519, 2-of-N steward) HPKE encapsulation key configuration, plus the OHTTP Relay that strips source IP before forwarding the encapsulated request to the STAR aggregator gateway and relays the encapsulated response back — both as one tier-aware `NexusApp` handler set, with TLS 1.3 + cert pinning + strict SNI + HSTS on every endpoint and a published key registry with expiry/rotation metadata. Select and pin the RFC-9458-default HPKE ciphersuite (`DHKEM(X25519, HKDF-SHA256) + HKDF-SHA256 + AES-128-GCM`) so S11's client encapsulation has a fixed contract. The deliverable is server-side standup ONLY — no client crypto (that is S11); the acceptance bar is "a live, signature-verifiable, IP-stripping relay endpoint exists and is reachable," which S11's client must existence-check before wiring.

**Acceptance criteria (testable):**

- EC-S10.1: the Key Configuration Server publishes a key config signed by ≥2 distinct Foundation steward keys; a config presented with only 1 valid steward signature is rejected by the verifying client path (2-of-N gate per `foundation-ops.md:79`). Behavioral test (call + assert) at `tests/integration/test_ohttp_key_config_steward_signed.py`.
- EC-S10.2 (the IP-strip is its only job, `foundation-ops.md:20`): an encapsulated request routed through the Relay arrives at the aggregator gateway with the client source IP absent/replaced; the aggregator never observes the originating IP, and the relay never observes the (encrypted) request body — the non-collusion split is verified on both axes (`tests/integration/test_ohttp_relay_ip_stripped.py`).
- EC-S10.3: relay-unreachable raises `OHTTPRelayUnavailableError` (`errors.py:39`, `foundation-health-heartbeat.md:52`) — the heartbeat is dropped for the cycle, counters retained; `OHTTPRelayDownError` (`foundation-ops.md:108`) surfaces the queue-locally-with-backoff path. Both typed failures assert distinctly.
- EC-S10.4: every Foundation OHTTP endpoint negotiates TLS ≥1.3 with the pinned cert and strict SNI; a TLS<1.3 or SNI-stripped handshake is refused (`network-security.md:15,24`, cross-checked against `tests/integration/test_strict_sni_enforcement.py`).
- EC-S10.5 (existence-check substrate for S11): a live read against the published key-registry endpoint returns a signature-valid, non-expired key config — this is the `gh api`-equivalent live receipt S11's client MUST hit before debugging access (`rules/verify-resource-existence.md` MUST-2). Wired at `tests/integration/test_ohttp_key_registry_live_fetch.py`.
- EC-S10.6 (Tor opt-in, default off): the Tor route is opt-in per-traffic-class; with Tor unselected the heartbeat path uses OHTTP-only (no Tor daemon dependency); `TorRouteUnavailableError` (`network-security.md:49`) surfaces only when Tor was explicitly requested and the daemon is unreachable (`tests/integration/test_tor_route_optional_phase02.py`). Default-on is NOT shipped (`network-security.md:77` item 4 resolved to opt-in).

**Capacity check (per `rules/autonomous-execution.md` § Per-Session Capacity Budget):** load-bearing logic ≈ the HPKE key-config signing/publish path + the relay IP-strip forwarding logic (~250-350 LOC); the Nexus handler set + key-registry `@db.model` is framework-stamped boilerplate. Live loop (deterministic OHTTP round-trip + Nexus integration harness). One shard. Three sentences: "Stand up the Foundation OHTTP Key Config Server publishing a 2-of-N-signed HPKE key config; stand up the IP-stripping Relay forwarding to the STAR aggregator; harden every endpoint with TLS 1.3 + pinned certs + strict SNI. **Implements:** `specs/foundation-health-heartbeat.md` design stack items 1-2 (§19-20 STAR + DP), cadence (§33-35 weekly, counters reset on send), error taxonomy `STARShardCorruptError`/`DPBudgetExceededError`/`kAnonymityFloorViolatedError`/`RitualCouplingDebounceTriggered`/`RandomIdRotationOverdueWarning` (§53-61); `specs/foundation-ops.md` STAR/Prio aggregator registry #5 (§21, Foundation-operated). **Scope:** Build the STAR **single-server** client (NOT Prio — settled `/todos` decision): each weekly report's per-counter measurement is split via `split_into_shares` into STAR threshold-shares keyed by the measurement value (identical measurements across clients produce combinable shares; below k the share is information-theoretically unrecoverable), encapsulated under S10's fetched+operator-signature-verified HPKE key (`OhttpClient.encapsulate_request` gated by `HeartbeatRegistryClient.fetch_aggregator_endpoint` + `verify_operator_signature`), and POSTed through the Relay. **Inject client-side / local DP per-counter BEFORE `split_into_shares`** (settled decision — holds against a compromised aggregator, matches the covert-channel-defense framing at `foundation-health-heartbeat.md:44`), with **per-metric published ε** tracked as a **total ε over the weekly reporting window** — per-counter ε does NOT compose across the 21 counters × M heartbeats, so the budget enforcement (`DPBudgetExceededError`, `errors.py:51`) drops the AFFECTED metric for the cycle while non-affected metrics report normally (per-metric budgets independent). **Pin the k≥100 floor to the TRUE cohort count, not the noised count:** STAR's threshold reveal operates on real distinct-submitter cardinality (the structural recovery condition is "≥k clients submitted the SAME measurement"); DP noise perturbs the per-counter VALUE a client reports, NOT the cohort-size cardinality the k-floor gates on — conflating the two would either leak (noised-down cohort revealed below true-100) or over-withhold. Honor the L-01 24h ritual-coupling debounce (`RitualCouplingDebounceTriggered`, `errors.py:63`) and per-install random-ID quarterly rotation (`RandomIdRotationOverdueWarning`, `errors.py:97`).
