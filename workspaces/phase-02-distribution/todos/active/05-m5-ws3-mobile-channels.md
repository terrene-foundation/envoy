## S13a — QR-pairing transcript-SAS handshake + threat-model device-pairing addition

**Type:** Build

**Value-anchor:** `ROADMAP.md:92` (source e — Phase-02 spec exit criterion the user approved) reads VERBATIM: "Flutter iOS + Android clients, QR-code pairing" + `ROADMAP.md:107` exit criterion VERBATIM: "Mobile QR-pairing works in <30s from cold start" (EC-02.6). This shard delivers the **authenticated** half of that pairing — the cryptographic handshake that makes "QR-pairing works" mean "a phone pairs to the user's local Envoy instance such that a network MITM cannot silently become a relay for every future Grant Moment", which is the security floor the headline onboarding moment rests on.

**Implements:** the device-pairing trust boundary identified as Spec gap #2/#3 in `01-analysis/01-research/03-ws3-mobile-channels.md` §1.2-1.3 + §"Spec gaps" #2/#3. Reuses: the Grant Moment nonce discipline (`specs/grant-moment.md:29` `nonce` + `GrantMomentReplayError` §145 as the replay-defense precedent → pairing gets its OWN single-use `pairing_nonce`); the `principal_genesis_id` sha256 anchor (`specs/connection-vault.md:31`); pairing-as-signed-Ledger-event modelled on `RuntimeAttestation`/`KeyRotationRecord` (`specs/runtime-abstraction.md` key-scope discipline). The phone enrolls as a **render-only terminal** — the local instance remains the signing authority; the phone does NOT hold the delegation key (`01-analysis/.../03-ws3-mobile-channels.md` §1.3 step 4).

**Sub-todo (REQUIRED — threat-model device-pairing addition):** add a `T-PAIR-*` device-pairing threat class to `specs/threat-model.md` (currently enumerates T-008/T-018/T-070/T-080 but no pairing threat — Spec gap #3). The addition MUST pin: (a) the **transcript-binding requirement** — SAS = truncated hash of the handshake transcript, NOT the visible secret (R1-CRIT-1); (b) the **≤2⁻²⁰ residual blind-relay MITM strength floor** (≥ ~7 numeric digits or equivalent rendering entropy); (c) the rendering format (pinned, both screens); (d) the QR-render-is-a-sensitive-surface mapping to screen-rec detection (handled in S13b). This is a spec ADDITION, not edit (per `rules/spec-accuracy.md` — describe what ships; the SAS handshake ships in THIS shard, so the threat-model entry resolves against working code at merge). Land the spec addition in the SAME PR as the handshake code per `rules/specs-authority.md` Rule 5.

**Depends:** S1 + S2a (WS-1 runtime contract + rs-bindings adapter — the AKE primitive runs over the FFI surface; the SPAKE2-vs-Noise-XX choice couples to S1's AKE conformance vectors and MUST be decided at launch).

**Scope:** Build the QR-pairing handshake as a short-authenticated-string (SAS) AKE bootstrapped by the QR, NOT a bearer-token-in-QR scheme. The local instance generates a fresh ephemeral X25519 keypair + a single-use high-entropy `pairing_nonce` + a `pairing_offer_expiry` (≤30s); the QR encodes `{principal_genesis_id, ephemeral_pubkey, pairing_nonce, pairing_offer_expiry, LAN/relay rendezvous}` and carries **no long-term secret and no bearer token**. The phone runs the AKE (SPAKE2 or Noise-XX with the QR nonce as channel-binding salt), both devices derive the SAS as a **truncated hash of the session-key transcript** (≥20 bits / ≤2⁻²⁰ residual MITM) and render it on BOTH screens for the user to compare; a mismatch (different transcript ⇒ MITM) aborts pairing; success emits a signed pairing Ledger event and enrolls the phone as a render-only terminal.

**Acceptance criteria (testable):**

- EC-S13a.1 (transcript-SAS mismatch aborts — the load-bearing correctness gate): a simulated active-network MITM that completes its own handshake leg with each side produces a **different transcript on each side** → the two rendered SAS values differ → the pairing flow raises a `PairingSASMismatchError` (or equivalent typed abort) and does NOT enroll the phone. Behavioral test (drive the handshake with an interposed relay, assert raise + assert no Ledger enrollment record + no keychain entry), at `tests/integration/test_qr_pairing_transcript_sas_mismatch.py`. A passing clean (no-MITM) handshake produces byte-identical SAS on both sides and enrolls successfully.
- EC-S13a.2 (SAS strength floor): the rendered SAS carries ≥20 bits of entropy from the transcript hash (≤2⁻²⁰ residual blind-relay MITM probability) — structural assert on the derived SAS bit-length / digit-count against the pinned rendering format, at `tests/integration/test_qr_pairing_sas_strength_floor.py`. The threat-model addition's strength-floor clause is the spec authority this asserts against.
- EC-S13a.3 (transcript-binding, NOT visible-secret): a test that holds the Trust-Vault visible secret CONSTANT while varying the transcript proves the SAS changes with the transcript (so the SAS is transcript-derived, not visible-secret-derived) — the structural defense against the R1-CRIT-1 regression. The visible secret MAY appear as familiar-UX framing but MUST NOT be the authenticating value.
- EC-S13a.4 (single-use expiring nonce — replay defense): a captured QR re-scanned after `pairing_offer_expiry` (≤30s) is refused (`PairingOfferExpiredError`); the same `pairing_nonce` presented twice is refused (reuses the `GrantMomentReplayError` nonce-discipline precedent). `tests/regression/test_tpair_qr_replay_refused.py`.
- EC-S13a.5 (render-only enrollment — no delegation key on device): a successful pairing enrolls a device-scoped credential keyed by `principal_genesis_id` but the phone receives **no** delegation/signing key; a structural assertion that the enrolled device record carries render-terminal capability only (the local instance remains signing authority). `tests/integration/test_qr_pairing_render_only_enrollment.py`.
- EC-S13a.6 (pairing is a signed Ledger event): a successful pairing emits a signed, auditable, revocable pairing record (modelled on the attestation/key-rotation record shape); re-pairing requires a deliberate re-issue (consistent with `connection-vault.md:113` no-transparent-sync). `tests/integration/test_qr_pairing_ledger_event.py`.
- EC-S13a.7 (threat-model addition resolves): the `specs/threat-model.md` `T-PAIR-*` addition exists and every claim in it (transcript-binding, ≤2⁻²⁰ floor, render format) resolves against the handshake code merged in this PR (per `rules/spec-accuracy.md` Rule 1 — no phantom citation).

**Capacity check (per `rules/autonomous-execution.md` § Per-Session Capacity Budget):** load-bearing logic ≈ the AKE handshake + transcript-SAS derivation + nonce/expiry discipline + render-only enrollment + Ledger-event emission (~350-450 LOC of dense cryptographic state-machine logic — this is the densest correctness surface in the milestone). Invariants held simultaneously: (1) SAS = transcript-hash NOT visible-secret, (2) ≤2⁻²⁰ strength floor, (3) single-use ≤30s nonce, (4) no-long-term-secret-in-QR, (5) render-only phone (no delegation key), (6) pairing-as-signed-Ledger-event = 6, within the 5-10 band. Loop = medium→live (the MITM-simulation harness IS the deterministic feedback loop once the AKE primitive is wired; before that, the SPAKE2-vs-Noise choice is a base-loop decision). One shard. Three sentences: "Build the QR-bootstrapped SAS handshake where the SAS is a truncated hash of the handshake transcript, not the visible secret. A MITM produces a different transcript → different SAS → user aborts; the phone enrolls render-only and pairing is a signed Ledger event. Add the matching T-PAIR threat-model entry pinning transcript-binding + the ≤2⁻²⁰ floor in the same PR."
