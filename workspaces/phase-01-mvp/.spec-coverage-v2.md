### HIGH-1 — `CrossChannelConfirmFailedError` raise path has no runtime-layer test

**Spec promise:** `specs/grant-moment.md § Error taxonomy` line 123 — error fires on "High-stakes Grant Moment cross-channel confirm leg failed."

**Code state:** raise site lives at `envoy/grant_moment/runtime.py:714-724`.

**Test state:** only constructed and asserted on shape at `tests/tier1/test_grant_moment_state_machine_transitions.py:576` (a wire-shape contract pin). NO test drives `submit_resolution` with `high_stakes=True` + non-Decline resolution + `cross_channel_confirmed=False`.

**Rule cited:** `rules/testing.md` § "MUST: Verify security mitigations have tests" — "For every § Security Threats subsection in any spec, grep for a corresponding `test_<threat>` function. Missing = HIGH." `CrossChannelConfirmFailedError` is the 10th member of the error taxonomy and the only one without a runtime-raise-path assertion.

**Fix:** add a regression test (e.g. `tests/regression/test_grant_moment_cross_channel_confirm_failed.py`) that:

5. Asserts `outcome.state == "ERROR"` AND `isinstance(outcome.error, CrossChannelConfirmFailedError)`

**Why fix in this PR:** the gap is same-bug-class as the H-03 test (one shard apart in the M3 sign-or-decline guard chain) and fits the `autonomous-execution.md` Rule 4 same-shard fix-immediately envelope (~30 LOC).

### HIGH-2 — Three public methods have ZERO call sites (orphan-detection)

**Spec promise:** Wave-4 runtime layer landed three new public methods (`emit_queue_hold`, `emit_queue_resume`, `novelty_class_for`) per `envoy/grant_moment/runtime.py:826, 842, 891` AND exported the two FRICTION*TOKEN constants in `__init__.py:150-152`. ### HIGH-3 — `DelegationRecord` entry-type tag vs payload-shape mismatch

**Spec promise:** `specs/ledger.md § Entry types` registers TWO distinct types:

- `DelegationRecord` — owner `specs/trust-lineage.md § DelegationRecord` (capability-delegation chain row)
- `grant_moment` — owner `specs/ledger.md § Ledger entry schemas §grant_moment` (per-action consent row)

**Code state:** `envoy/grant_moment/runtime.py:167` defines `_ENTRY_DELEGATION_RECORD = "DelegationRecord"`. The runtime emits this entry_type with payload shape matching `grant_moment` (request_id, result_id, decision, phase_a_ref, intent_id, modify_payload, author_payload, co_signer_principal_genesis_id, signatures) — NOT the trust-lineage.md DelegationRecord shape (delegation_id, chain_parent_id, delegator/delegatee, capabilities, valid_from/until, algorithm_identifier).

**Rule cited:** `rules/specs-authority.md` Rule 5b — sibling-spec re-derivation. Edit of `grant-moment.md` triggers sweep against ledger.md + trust-lineage.md. Both sibling specs describe shapes the runtime contradicts.

**Fix:** the cleanest fix is to retag — change `_ENTRY_DELEGATION_RECORD = "DelegationRecord"` to `_ENTRY_GRANT_MOMENT = "grant_moment"`, and update payload fields to match ledger.md:354-364 exactly (rename `request_id` → `request_ref`, `result_id` → `result_ref`, add `envelope_version_at_decision`, `novelty_class`, `signed_by: "delegation_key"`). Tests asserting `action == "DelegationRecord"` (e.g. `tests/integration/test_grant_moment_state_machine.py:168-171`) MUST update in same commit per `rules/orphan-detection.md` Rule 4 (API removal sweeps tests).

If retag is too invasive for one shard, the alternative is to keep the type tag `DelegationRecord` BUT augment the payload with the trust-lineage.md fields (delegation_id, chain_parent_id, delegator, delegatee, capabilities, valid_from, valid_until, algorithm_identifier). This is wider scope (sibling Trust Lineage shard work).

**Why fix in this PR:** type-tag-vs-shape drift is the institutional ratchet that lets downstream verifiers either reject the row (wrong shape) or mis-interpret it (wrong type semantics). The Independent Verifier (EC-4) builds an audit trail against the entry-type registry; an entry typed `DelegationRecord` but shaped `grant_moment` fails the verifier.
