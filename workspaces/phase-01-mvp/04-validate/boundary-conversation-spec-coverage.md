## Findings

### HIGH-1 — No production path clears the Shamir suspension; real users gate-lock at S8→S9

**File:** `envoy/boundary_conversation/runtime.py:436-462` (set) + `479-485` (gate-back read); no clear-path anywhere in `envoy/`.

The S8 handler sets `plan.suspension` and the S9 Gate 2 refuses to sign while `plan.suspension is not None`, forcing the user back to S8. But **no production method clears the suspension**. `resume()` (line 192) rehydrates from Trust Vault via `Plan.from_dict`, which round-trips the suspension intact (verified empirically), so resuming RESTORES the block. The analysis doc § 3.3 + § 5.5 specify the clear-path as a user-invoked `envoy boundary resume <ritual_id>` that checks `ShamirRitualCoordinator.is_complete(...)` then clears — that method does not exist. Every passing test masks the gap by reaching into runtime internals: `runtime.current_plan(ritual_id).suspension = None` (`test_boundary_conversation_runtime_wiring.py:248`, `test_boundary_conversation_pause_resume_shamir.py:191`, `test_boundary_conversation_per_state_ledger_entries.py:201`). A real first-time user who completes the physical Shamir card distribution has no API to advance past S8 → they can never sign an envelope → EC-1 (N=3 ≤25min) is unreachable on the happy path. This is the orphan/half-implemented-contract pattern (`rules/orphan-detection.md`, `zero-tolerance.md` Rule 6): the suspension-SET and gate-back-READ shipped, the suspension-CLEAR did not. Disposition: add the production clear-path (e.g. `resume_from_shamir(ritual_id)` that verifies `shamir_coordinator.is_complete(...)` then sets `plan.suspension = None` + `plan.state = PlanState.DRAFT`), with a Tier 2 test that drives the clear through the public surface rather than poking `current_plan(...).suspension`. T-02-43 step 1 nominally owns "S8 Shamir pause", but since the SET + gate-back already shipped on this branch, the matching CLEAR fits the same shard budget and should land here (`autonomous-execution.md` Rule 4).

### MED-1 — Assembler `_FED_STATES` feeds S6/S7/S8 extractions that no dimension builder consumes

**File:** `envoy/boundary_conversation/envelope_assembler.py:49-60, 170-233`.

`_FED_STATES` includes `S6_template_offer`, `S7_visible_secret`, `S8_shamir`, so `feed()` records their extractions. But the five `_build_*` dimension builders only synthesize constraints from S1–S5 (`_authored_constraints_for("S1_money")`…`"S5_first_task"`). The S6/S7/S8 extractions are recorded into `self._extractions`, persisted via `to_dict()`, but never assembled into any dimension — silently dropped from the envelope. The analysis doc § 3.7 says S6/S7/S8 "feed the runtime (template cache / Trust Vault / backup), NOT the envelope dimensions" — which means they should NOT be in `_FED_STATES` at all (the assembler is the envelope producer, not the runtime). Recording-then-dropping is a Rule-3c-adjacent silent-no-effect: data is accepted but has zero effect on output. Disposition: either narrow `_FED_STATES` to S1–S5 (cleanest — matches § 3.7 intent), or document why S6/S7/S8 are recorded for round-trip but excluded from assembly. Low blast radius (no incorrect envelope, just dead recording) → MED not HIGH.
